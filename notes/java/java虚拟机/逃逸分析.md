# 1、什么是逃逸分析

逃逸分析是 JIT 编译器的一种优化手段，核心就是判断一个对象会不会“跑出”当前方法或当前线程。如果对象老老实实待在方法内部，不被外界引用，JVM 就能放心地做一些激进优化。



# 2、逃逸分析两种程度：

1）方法逃逸：对象被 return 出去了，或者作为参数传给别的方法，总之跑出了当前方法的作用域。

2）线程逃逸：对象被赋值给静态变量、实例变量，或者被其他线程访问到了，这种逃逸程度最高。



# 3、根据逃逸分析的结果，JVM 能做三种优化：

## 1）栈上分配：

对象没逃逸，直接在栈帧里分配，方法一结束内存就跟着回收了，压根不用 GC 操心。

> 严格来说，HotSpot 虚拟机目前并没有真正实现栈上分配。它用的是标量替换来间接达到“不在堆上分配”的效果。原因是 HotSpot 的对象内存布局和 GC 配合得很紧密，真要在线程栈上分配完整对象，改动太大，收益又不明显。
>
> 所以面试时说“栈上分配”，其实背后干活的都是标量替换。对象不是整个搬到栈上，是被拆散成标量放到栈上或者直接放寄存器里。

## 2）标量替换：

如果对象没逃逸，JVM 干脆把对象“打散”，把里面的字段当成独立的局部变量来用，连对象头都省了。

**标量替换的底层机制：**

标量就是不可再分的数据，比如 int、long、引用地址这些。与之对应的是聚合量，比如一个对象，里面包含多个字段。

```java
public void calculate() {
    Point p = new Point();
    p.x = 10;
    p.y = 20;
    int sum = p.x + p.y;
}

```

如果 Point 没逃逸，JVM 会把上面的代码优化成这样：

```java
public void calculate() {
    int x = 10;
    int y = 20;
    int sum = x + y;
}

```

Point 对象压根没创建，省掉了对象头那 12 字节的开销，也省掉了堆分配和后续 GC 的负担。这在循环里创建大量临时对象的场景下效果特别明显，比如每秒创建几百万个坐标点对象，优化后这些对象全变成栈上的基本类型变量了。

## 3）同步消除：

对象只在单线程内使用，不会被其他线程看到，那加在上面的 synchronized 就是多余的，直接去掉。

# 4、逃逸分析的触发条件

逃逸分析只在 JIT 编译阶段生效，解释执行是没有这个优化的。要想触发 JIT 编译，代码得“热”起来：

1. 方法被调用超过一定次数（Client 模式默认 1500 次，Server 模式默认 10000 次）

2. 循环体执行次数超过阈值，触发 OSR 编译

可以通过 JVM 参数控制逃逸分析：

```shell
-XX:+DoEscapeAnalysis     # 开启逃逸分析，JDK 6u23 后默认开启
-XX:-DoEscapeAnalysis     # 关闭逃逸分析
-XX:+EliminateAllocations # 开启标量替换，依赖逃逸分析
-XX:+EliminateLocks       # 开启同步消除，依赖逃逸分析
-XX:+PrintEscapeAnalysis  # 打印逃逸分析结果（debug 版本 JVM）

```

# 5、逃逸分析的局限性

逃逸分析本身是有开销的，分析过程需要遍历方法的控制流图。如果方法特别复杂、调用链特别深，分析的成本可能比优化带来的收益还大。JVM 会做权衡，对于太复杂的方法可能直接放弃分析。

还有一点，逃逸分析是保守的。只要有一丁点可能逃逸，就按逃逸处理。比如对象被传给一个没有被内联的方法，即使那个方法实际上没把对象传出去，JVM 也会认为它逃逸了，因为看不到那个方法的内部实现。

# 6、常见问题

- **提问：逃逸分析开启后，是不是所有局部对象都会在栈上分配？**

回答：不是。首先 HotSpot 没有真正的栈上分配，用的是标量替换。其次能被优化的对象必须满足几个条件：对象没有逃逸、对象可以被拆解成标量、代码已经被 JIT 编译。如果对象太大，或者包含不能拆解的字段，或者代码还在解释执行阶段，都不会触发这个优化。

- **提问：怎么验证逃逸分析是否生效了？**

回答：可以用 -XX:+PrintEscapeAnalysis 打印分析结果，不过这个参数只有 debug 版本的 JVM 才有。更实用的办法是对比开启和关闭逃逸分析时的 GC 日志，如果优化生效，年轻代 GC 频率会明显下降。也可以用 JITWatch 工具看 JIT 编译日志，里面会显示哪些对象被标量替换了。

- **提问：同步消除能消除哪些锁？**

回答：只能消除那些被证明不会逃逸的对象上的锁。比如方法内部 new 了一个 StringBuffer 然后加锁操作，这个 StringBuffer 没传出去，那锁就能被消除。但如果是锁在 this 上，或者锁在传递来的参数上，JVM 没法确定不会被其他线程访问，就不会消除。实际效果是，那些多余的 synchronized 方法调用（比如单线程用 StringBuffer）会被优化掉。
